# Contributing to Hestia Foundation

Last updated: 2026-05-29
For developers, builders, researchers, writers, and anyone who wants to help

---

## How to Contribute

### 1. Build and Document
The most valuable contribution: build a Hestia Shell A (or variation), document it, and share your experience.

**What to document:**
- Photos and videos of every stage
- Actual material costs (with receipts)
- Time tracking by task
- Problems encountered and solutions found
- Modifications you made to the system
- Performance data (temperature, humidity, energy use)

**How to share:**
- Open a GitHub issue with your build report
- Email info@hestiafoundation.org with a link to your documentation
- Post on social media with #HestiaFoundation

### 2. Improve the Build Manual
Our build manual is a living document. If you find:
- Unclear instructions
- Missing steps
- Outdated information
- Better techniques

Please submit a correction or addition via GitHub pull request or email.

### 3. Translate
We need translations into:
- Portuguese (priority — we are based in Portugal)
- Spanish
- French
- German
- Any other language where affordable housing is needed

**How to translate:**
1. Pick a document from the repo
2. Create a translated version with language suffix (e.g., `FAQ_pt.md`)
3. Submit via pull request or email

### 4. Research and Data
- Climate data for different regions
- Local material costs in different countries
- Building code requirements by jurisdiction
- Structural analysis or thermal modelling
- Life cycle assessment (LCA)
- Social housing policy research

### 5. Software and Tools
- CAD improvements (FreeCAD, SketchUp, Blender)
- Cost calculators or web apps
- Sensor data visualisation dashboards
- Mobile apps for build teams
- Translation tools or workflows

### 6. Design Variations
- Adaptations for different climates (cold, tropical, humid)
- Different sizes (smaller or larger than 37.5 m²)
- Multi-storey adaptations
- Different foundation systems
- Interior design layouts
- Furniture designs using pallets

### 7. Community Building
- Organise local build workshops
- Start a Hestia meetup in your city
- Write blog posts or make videos about the project
- Connect us with potential partners (universities, municipalities, NGOs)
- Spread the word on social media

## GitHub Workflow

### For Small Changes (typos, single-file updates)
1. Fork the repository
2. Edit the file in your fork
3. Submit a pull request with a clear description

### For Larger Contributions (new documents, design variations)
1. Open an issue first describing what you want to add
2. Discuss with the team (we respond within 1 week)
3. Fork and create a feature branch
4. Submit a pull request referencing the issue

### File Naming Conventions
- Use lowercase with underscores: `build_manual_v2.md`
- Add language suffix for translations: `FAQ_pt.md`, `BUILD_es.md`
- Add version number for major revisions: `build_protocol_v2.md`

## Code of Conduct

### Our Standards
- **Be respectful.** Treat everyone with respect regardless of experience, background, or identity.
- **Be constructive.** Criticise ideas, not people. Offer solutions, not just problems.
- **Be open.** Share knowledge freely. The goal is collective progress, not individual credit.
- **Be honest.** Document failures as thoroughly as successes. Misleading data helps no one.
- **Be safe.** Never compromise on safety for speed or cost. Construction can be dangerous.

### Unacceptable Behaviour
- Harassment, discrimination, or intimidation of any kind
- Sharing private information without consent
- Trolling, insulting/derogatory comments, or personal attacks
- Publishing misleading or fabricated data
- Ignoring safety guidelines in build documentation

### Enforcement
Violations may result in:
1. Warning and request to stop
2. Temporary or permanent ban from the project
3. Removal of contributions if they violate safety or accuracy standards

Report violations to: info@hestiafoundation.org

## Decision-Making

Hestia Foundation is currently small and founder-led. As we grow, we will formalise governance:

**Current (2026):**
- Founder (Sebastian Brosche) makes final decisions
- AI assistant (Kimi Claw / Miha) coordinates and documents
- Contributors influence decisions through discussion and pull requests

**Future (post-registration):**
- General Assembly (all members) elects Board of Directors
- Board makes strategic decisions
- Technical decisions made by build team and advisors
- Financial decisions made by Board with Fiscal Council oversight

**For now:** If you have a strong opinion on project direction, open an issue or email us. We read everything.

## Attribution

When you contribute, you will be:
- Listed in the GitHub repository contributors
- Credited in relevant documents (e.g., "Build manual updated by [Name]")
- Named in annual reports and publications (with your consent)
- Invited to milestone events and celebrations

We do not offer monetary payment for contributions (we are a non-profit with limited funds). We offer recognition, learning, and the satisfaction of building something meaningful.
